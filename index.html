Harness Engineering: от 50 строк Python до production-агента

Курс построен по материалам репозитория nexu-io/harness-engineering-guide. Цель: понять, как harness превращает LLM в агента с циклом действий, инструментами, памятью, контекстом, sandbox и guardrails.

Для кого

Для разработчиков, которые уже умеют писать backend-код и хотят понять, как устроены coding agents, managed agents и agent runtimes без магии фреймворков.

Результат

После курса у вас будет рабочий harness: tool loop, registry, memory files, context budget, permission checks, sandbox plan и eval checklist.

Формат

Каждый модуль: чтение исходной статьи, короткий конспект, практическое задание и чеклист проверки. Прогресс сохраняется в браузере.

Карта курса

Главная идея репозитория: ценность агента живет не только в модели, а в обвязке вокруг нее. Harness отвечает за то, что модель видит, какие действия может выполнять и как система восстанавливается после ошибок.

Модули

Модуль 1. Что такое harness и почему это не просто framework
60-75 минконцептыархитектура

Harness - это runtime-обвязка, которая превращает модель в агента: собирает контекст, показывает доступные инструменты, выполняет tool calls, хранит состояние и применяет guardrails. Фреймворк помогает строить harness, но не заменяет инженерные решения вокруг безопасности, памяти и жизненного цикла.

  • what-is-harness.md
  • harness-vs-framework.md
  • comparison.md
Модуль 2. Первый harness в 50 строк
90 минкодPython

Минимальная версия состоит из system prompt, JSON-схем инструментов, функции dispatch и цикла: вызвать модель, выполнить tool calls, вернуть результаты в историю, повторить до финального ответа.

for turn in range(MAX_TURNS):
    msg = llm(messages, tools)
    messages.append(msg)
    if not msg.tool_calls:
        return msg.content
    for call in msg.tool_calls:
        messages.append(run_tool(call))

Модуль 3. Agentic loop: выходы, бюджеты, параллельные tool calls
75 миннадежность

Production loop не просто повторяет вызовы модели. Он умеет останавливаться, распознавать зацикливание, обрабатывать tool errors как наблюдения, сохранять trace и не засорять контекст огромными результатами инструментов.

Модуль 4. Tool system: registry, schemas, dynamic skills
90 минtoolsMCP

Модель видит только схему инструмента, а harness выполняет реальный handler. Качество описаний, единый формат ошибок и динамическая загрузка skills часто важнее, чем выбор модели.

  • tool-system.md
  • skill-system.md
Модуль 5. Memory & context: что модель видит в каждом turn
90 минcontext engineering

Context - это один API-вызов, session - состояние задачи, memory - то, что переживает перезапуск. Context assembler решает упаковочную задачу: system prompt, tools, task, memory, файлы и недавняя история должны поместиться в бюджет с резервом на ответ.

  • memory-and-context.md
  • context-engineering.md
Модуль 6. Guardrails и sandbox: модель просит, harness решает
90 минsecurity

Guardrails нельзя сводить к промпту. Проверка прав должна выполняться в коде перед каждым действием: allow/deny/modify/ask. Sandbox ограничивает последствия на уровне ОС: filesystem, network, process, user permissions.

  • guardrails.md
  • sandbox.md
  • classifier-permissions.md
Модуль 7. Sub-agents, orchestration и long-running tasks
75 минmulti-agent

Sub-agents полезны, когда нужна изоляция контекста, специализация или параллельное исследование. Но orchestration стоит дорого: нужен timeout, bounded fan-out, понятный формат результата и защита от shared mutable state.

  • sub-agent.md
  • multi-agent-orchestration.md
  • long-running-harness.md
  • managed-agents-architecture.md
Модуль 8. Error handling, scheduling и evals
90 минproduction

Надежный harness классифицирует ошибки, ретраит только временные сбои, умеет checkpoint/resume, запускает scheduled work без загрязнения основной сессии и оценивает агентов с учетом инфраструктурного шума.

Практические работы

Практикумы можно выполнять последовательно в одном mini-harness проекте.

Lab 1. File agent

Соберите read/write file assistant с MAX_TURNS, trace log и явными ошибками инструментов.

Lab 2. Context budget

Добавьте ContextAssembler с приоритетами и тестом, который доказывает, что low-priority секции отбрасываются первыми.

Lab 3. Permissions

Вставьте guardrail layer перед dispatch: read разрешен, write логируется, shell требует approval stub, dangerous commands запрещены.

Мини-тест

1. Где должна исполняться реальная проверка прав?

2. Что модель видит при tool use?

3. Почему нужен MAX_TURNS?

Итоговый проект

Соберите mini-harness для исследовательского агента, который принимает URL репозитория, читает README и выбранные файлы, строит план изучения и сохраняет HTML-курс. Это ровно тот тип задачи, для которой нужны loop, tools, context, memory и guardrails.

Минимальные требования

Критерий готовности: другой разработчик может запустить проект, пройти курс и увидеть, какие решения в harness являются учебными, а какие обязательны для production.
