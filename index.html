Карта курса

Курс идет от минимального harness к production-архитектуре. Главная идея репозитория: framework помогает быстро стартовать, но harness дает контроль над инструментами, памятью, безопасностью, бюджетами и жизненным циклом агента.

Foundation

Что строим

Harness как runtime wrapper: он исполняет tools, собирает context, ведет memory, ставит guardrails и держит agentic loop.

Engineering

Как управляем

Циклы, бюджеты ходов, parallel tool calls, context priority, skill loading, sandbox и retry/degradation стратегии.

Production

Как масштабируем

Sub-agents, orchestration, scheduling, managed architecture, eval infrastructure, permission classifiers и agent teams.

Архитектура harness

Модули

Открывай модули по порядку. Ссылки ведут на исходные markdown-файлы в репозитории.

1. Ментальная модель harness
what-is-harness harness-vs-framework comparison

Разберись в различии model, harness, framework и runtime. Хороший harness не прячет важные решения: он делает их явными и тестируемыми.

2. Первый рабочий агент
your-first-harness agentic-loop

Собери минимальный цикл: system prompt, tool schema, tool execution, observation append, exit conditions. Главный риск начинающих - бесконечная петля без бюджета и критериев остановки.

while turns_left:
    response = model(messages, tools=registry.schemas())
    if response.final:
        break
    results = run_tool_calls(response.tool_calls)
    messages.append(observations(results))
3. Tool system и skill system
tool-system skill-system

Tools - исполняемые возможности. Skills - лениво подгружаемые инструкции и наборы инструментов. Практичный паттерн: thin harness, thick skills, где harness знает как грузить и изолировать способности, а domain logic живет рядом со skill.

4. Context engineering и memory
context-engineering memory-and-context

Контекст - не свалка. Нужна приоритизация: system, user task, repo instructions, recent messages, retrieved facts, summaries. Memory делится на raw daily logs и curated long-term memory.

5. Guardrails, sandbox и permissions
guardrails sandbox classifier-permissions

Безопасность строится слоями: trust boundary, allow/deny/tiered permissions, isolated execution, network/filesystem limits и classifier-based permission checks для снижения approval fatigue.

6. Ошибки, retry и graceful degradation
error-handling

Хороший harness классифицирует ошибки: transient, permission, validation, model, infrastructure. Retry нужен не везде; иногда правильнее деградировать, запросить данные или checkpoint/resume.

7. Sub-agents и orchestration
sub-agent multi-agent-orchestration agent-teams

Sub-agents нужны для изоляции контекста, специализации и параллелизма. Изучи pipeline, fan-out/fan-in, supervisor и git-based coordination для больших задач.

8. Долгоживущие и managed agents
long-running-harness managed-agents-architecture initializer-coding-pattern

Долгие задачи ломаются от context anxiety и self-evaluation bias. Репозиторий предлагает reset vs compaction, generator-evaluator архитектуру и разделение brain/session/hands.

9. Scheduling, automation и heartbeats
scheduling-and-automation

Автоматизация бывает one-shot, recurring cron, event-triggered и heartbeat-based. Важно выбрать session targeting, delivery surface и payload contract, чтобы задача не теряла контекст.

10. Evals, инфраструктурный шум и реальные кейсы
eval-infrastructure eval-awareness nexu-windows-packaging ghost-account-hunting

Agent evals чувствительны к ресурсам, времени, изоляции и тому, распознает ли агент тест. Производственные кейсы показывают, что harness engineering - это не только LLM code, но и инфраструктура вокруг него.

Финальный проект

Собери Mini Harness для coding assistant

Цель: сделать маленький, но честный harness, который читает задачу, собирает контекст из папки проекта, может читать файлы, запускать безопасные команды и возвращать итог с логом действий.

Обязательные компоненты

  • Agentic loop с лимитом ходов и явным stop condition.
  • Tool registry минимум из трех tools: read_file, list_files, run_tests.
  • Context assembler с приоритетами и ограничением бюджета.
  • Permission policy: allow для чтения, ask/deny для опасных shell-команд.
  • Session log и краткий final report.

Критерии защиты

  • Можно объяснить, где проходит trust boundary.
  • Ошибки tools классифицируются и не ломают весь loop.
  • Есть один eval сценарий с ожидаемым результатом.
  • Документация показывает, когда добавлять skill или sub-agent.
